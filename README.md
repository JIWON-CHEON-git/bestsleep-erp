# 베스트슬립 재고흐름 ERP

매트리스 브랜드 "베스트슬립"의 주문 데이터를 가지고, 주문 → 재고 차감 → 재주문 발주 → 입고 → 밀린 주문 출고로 이어지는 재고 흐름 전체를 날짜별로 시뮬레이션하고, 그 결과를 운영자 화면으로 보여주는 ERP 데모입니다.

- **백엔드**: Django 4.2 + DRF (SQLite)
- **프론트엔드**: Next.js 15 (App Router) + TypeScript + Tailwind + shadcn/ui

```
주문 접수 → 재고 차감(출고) → 재고가 재주문점 이하 → 생산발주
→ (며칠 뒤) 입고 → 밀려 있던 주문 출고 → 고객 도착
```

시뮬레이션 결과는 DB에 저장되고, 화면에서는 **기준일(오늘 날짜)** 을 선택하면 그 시점까지의 상황만 보여줍니다. 예를 들어 기준일을 6/10로 두면 6/10까지 들어온 주문과 6/10까지 일어난 일만 보입니다.

화면은 3개입니다.

| 화면 | 경로 | 한 줄 설명 |
|---|---|---|
| 재고 현황 | `/inventory` | 제품별 재고 상태 + 선택한 제품의 날짜별 재고 변화 |
| 생산발주 목록 | `/production-orders` | 언제 무엇을 얼마나 생산 걸었고, 언제 들어오는지 |
| 배송 안내 | `/orders` | 주문이 언제 나가고 언제 도착하는지, 늦어지면 왜 늦어지는지 |

## 시작하기

### 백엔드

```bash
cd backend                                                    # 백엔드 폴더로 이동
python -m venv venv                                           # 가상환경 생성
venv\Scripts\activate                                         # 가상환경 활성화 (Windows)
# source venv/bin/activate                                    # 가상환경 활성화 (macOS / Linux)
pip install -r requirements.txt                               # 패키지 설치
python manage.py migrate                                      # DB 테이블 생성
python manage.py seed_data --excel ../data/exampledata.xlsx  # 제품·주문 데이터를 DB에 적재
python manage.py run_simulation                               # 재고 흐름 시뮬레이션 실행 (발주·배송 결과 계산)
python manage.py runserver 8000                               # 백엔드 서버 실행 (localhost:8000)
```

> **Windows에서 한글이 깨질 때**: 위 명령은 전부 엑셀 → ORM 경로로 데이터를 쓰기 때문에 한글이 깨지지 않습니다. 다만 `python manage.py loaddata <fixture>`처럼 JSON 픽스처를 직접 불러올 때는 Windows 콘솔 기본 인코딩 때문에 한글이 깨질 수 있습니다. 이 경우 실행 전에 아래를 먼저 입력하세요.
>
> ```powershell
> $env:PYTHONUTF8=1     # PowerShell
> set PYTHONUTF8=1      # cmd
> ```

### 프론트엔드

```bash
cd frontend                                                   # 프론트엔드 폴더로 이동
npm install                                                   # 패키지 설치
npm run dev                                                   # 프론트엔드 서버 실행 (localhost:3000)
```

브라우저에서 http://localhost:3000 접속 (자동으로 `/inventory`로 이동합니다)

### 테스트

```bash
cd backend
pytest
```

## 화면 구성

모든 화면은 헤더의 **기준일 선택기**를 공유합니다. 기준일을 바꾸면 그 날짜 기준으로 재고/발주/주문 상태가 다시 계산되어 보입니다.

### 재고 현황 (`/inventory`)

- **상단 요약**: 재주문점 이하 제품 수, 재고 부족 제품 수, 진행 중인 생산발주 수, 지연 중인 주문 수
- **제품 목록**: SKU, 제품명, 카테고리/사이즈, 현재고, 가용재고, 안전재고, 재주문점, 입고예정 물량, 백오더, 상태(정상/주의/발주중/부족). 행을 클릭하면 아래에 그 제품의 날짜별 흐름이 표시됩니다. 재주문점 이하 행은 노란색, 가용재고가 음수인 행은 빨간색 보더로 강조됩니다. 상태별 필터, 재고 컬럼 정렬을 지원합니다.
- **일자별 원장**: 선택한 제품의 하루 단위 변화를 봅니다. 기초재고 → 생산입고(+) → 주문출고(−) → 기말재고, 그리고 가용재고와 그날 발생한 생산발주 수량, 백오더 잔량, 이벤트를 보여줍니다. 생산발주가 발생한 날은 파란색, 재고가 부족했던 날은 빨간색/주황색 보더로 구분됩니다. 같은 데이터를 그래프로도 보여줍니다.

> **"현재고"와 "가용재고"는 다릅니다.** 현재고는 지금 창고에 있는 실제 수량이고, 가용재고는 거기에 "입고될 생산발주분"을 더하고 "이미 약속된(밀린) 주문"을 뺀 수량입니다. 재주문이 필요한지는 가용재고로 판단합니다.

### 생산발주 목록 (`/production-orders`)

- 발주일, SKU, 제품명, 발주수량, 입고예정일, 상태(입고 예정 / 입고 완료), 발주 사유
- 발주 사유는 어떤 주문을 처리한 직후에 재주문점에 도달했는지를 함께 보여줍니다. 예: `ORD-0007 처리 후 가용재고 2개 → 재주문점(12) 이하`
- 상태·제품·발주일로 필터, 입고예정일 기준 정렬을 지원합니다.

### 배송 안내 (`/orders`)

- 주문번호, 주문일, 제품, 수량, 고객명, 희망배송일, 출고일, 도착예정일, 상태, 비고, 취소
- 상태 종류: 배송 대기 / 정상 배송 / 배송 지연(희망배송일 없이 늦어짐) / 기한 내 배송(희망배송일을 지킴) / 기한 초과(희망배송일을 못 지킴) / 주문 취소. 지연이 있는 주문은 상태 아래에 지연일수가 함께 표시됩니다.
- 비고에는 지연·기한초과 주문의 원인을 발주-입고 흐름으로 보여줍니다. 예: `생산 입고 대기 (06-04 발주 → 06-11 입고, 1일 초과)`. 아직 백오더 상태로 입고 자체를 기다리는 주문은 `입고 대기 중`으로 표시됩니다.
- 지연·대기·기한초과 주문은 행 색으로 강조됩니다.
- 아직 출고 전인 주문은 "취소" 버튼으로 취소할 수 있습니다. 취소하면 재고/원장에 미친 영향도 되돌립니다.

## 설계 결정

### 부분 출고 금지

재고가 모자라면 일부만 먼저 보내지 않고 전량을 대기열(백오더)에 둡니다. 매트리스·가구는 부분 배송이 오히려 불편을 주는 품목이라, 한 번에 모아 보내는 쪽이 운영에도 고객 경험에도 낫다고 봤습니다.

### 희망배송일 처리

여기가 가장 오래 고민한 부분입니다. 재고를 특정 주문에 배정하는 일(재고 차감)은 주문이 들어온 순서(FIFO) 그대로 진행하고, 트럭이 창고에서 실제로 나가는 시점만 `희망배송일 - 배송 리드타임`에 맞춰 따로 늦춥니다.

처음에는 "재고가 있으면 바로 보내는 게 맞지 않을까" 싶었습니다. 그런데 침대·매트리스는 이사 일정에 맞춰 주문하는 경우가 많아서, 너무 일찍 도착하는 것도 늦게 도착하는 것만큼 불만으로 이어질 수 있습니다. 그래서 "재고 확보"와 "배송 출발"을 분리하기로 했습니다. 재고 확보는 FIFO를 그대로 따르게 두고, 물리적인 출발 시점만 늦추는 식입니다. 이렇게 하면 "이 주문 몫의 재고는 이미 빼놨지만 아직 트럭에는 안 실었다"는 상태가 자연스럽게 표현되고, 뒤에 들어온 일반 주문이 먼저 출고되는 일도 없습니다.

### 밀린 주문 처리 순서

대기 중인 주문은 들어온 순서대로 처리합니다. 맨 앞 주문이 막히면, 뒤 주문 수량이 더 작아서 채울 수 있어도 건너뛰지 않고 같이 기다립니다. 먼저 주문한 사람이 먼저 받아야 한다는 원칙이 더 중요하다고 판단했고, 순서를 건너뛰는 로직은 "왜 내 주문만 뒤로 밀렸는지"를 설명하기도 까다롭습니다.

### 발주 수량 및 연속 발주

생산발주는 항상 MOQ 단위로 나갑니다. 한 번 발주해도 가용재고가 재주문점을 못 넘으면, 넘을 때까지 같은 날 여러 건을 이어서 발주합니다. 대량 주문으로 재고가 한 번에 크게 빠지는 상황에서도 발주 로직이 멈추지 않고, 회복 가능한 수준까지 알아서 따라가게 하고 싶었기 때문입니다.

### 주문 취소 가능 여부

취소 가능 여부는 전체 계산 결과가 아니라 기준일(오늘) 시점으로 판단합니다. 출고일이 기준일보다 이전이면 이미 벌어진 일로 보고 취소를 막고, 기준일 이후라면 아직 막을 수 있는 주문으로 봅니다.

전체 계산 결과만 보면 "이미 출고됨"으로 표시되는 주문도, 운영자 입장에서는 아직 출고 전인 미래 일정일 수 있습니다. 기준일 재해석을 취소 판단에도 똑같이 적용해서, 화면에 보이는 정보와 실제로 할 수 있는 행동이 어긋나지 않게 했습니다.

### 생산·배송 캐파 무제한 가정

같은 날 생산발주나 출고가 몰려도 라인·차량 부족으로 인한 추가 지연은 계산하지 않습니다. 필요하면 외주로 처리한다고 가정했고, 제품마다 캐파가 독립적이라고 본 단순화입니다.

### 기준일 동작 방식

시뮬레이션 자체는 전체 기간을 한 번에 계산해서 DB에 저장해 둡니다. 화면에 보여줄 때는 기준일을 "오늘"로 놓고, 그 시점까지 일어난 일만 보이도록 결과를 다시 해석하는 레이어를 하나 더 둡니다. 기준일 이후의 주문·생산발주·원장 데이터는 화면에 노출되지 않습니다.

전체 계산 결과를 그대로 보여주면 아직 일어나지 않은 미래의 일까지 미리 알 수 있게 되어버려서, 운영자가 실제로 알 수 있는 정보와 화면에 보이는 정보가 어긋납니다. 그래서 아래 항목들은 모두 기준일 기준으로 다시 계산합니다.

- **현재고/가용재고**: 날짜가 기준일 이하인 가장 최근 원장 행의 값을 사용합니다. 기준일이 시뮬레이션 시작일보다 이전이면 초기재고를 그대로 사용하고, 종료일 이후면 마지막 행 값으로 고정됩니다.
- **입고예정 수량·진행 중 생산발주**: `발주일 ≤ 기준일 < 입고예정일` 조건으로 "발주는 됐지만 아직 입고 전"인 건을 계산합니다.
- **생산발주 상태**: 입고예정일과 기준일을 비교해 "입고 예정/입고 완료"를 요청마다 다시 계산합니다.
- **주문 상태**: 재고 차감일이 기준일보다 미래(또는 아직 없음)이면 "대기 중"으로 보고 예상 입고일을 계산합니다. 이미 재고가 차감된 주문은 시뮬레이션이 계산한 최종 결과를 그대로 보여줍니다 — 이미 일어난 일은 기준일과 무관하게 확정적이기 때문입니다. 주문일이 기준일보다 미래인 경우도 같은 "대기 중" 분기를 타지만, 사유는 "주문일 이후 처리될 예정"으로 다르게 표시합니다.

### 데이터 확장성

시뮬레이션은 호출될 때마다 DB에 있는 주문 전체를 다시 읽어 처음부터 다시 계산하고, 기존 결과는 지운 뒤 새로 채웁니다. 지금 규모(제품 10종, 주문 수십 건)에서는 재계산 비용이 무시할 만해서 가장 단순하고 정확한 방식을 택했습니다. 데이터가 훨씬 커지면 이전 상태를 이어받아 하루치만 계산하는 증분 방식으로 바꿔야겠지만, 계산 로직이 이미 하루 단위 함수로 나뉘어 있어서 전환 자체는 크게 어렵지 않을 것 같습니다.

## 계산 로직

제품마다 시작일부터 종료일까지 하루씩 진행하며 아래 순서로 계산합니다.

1. **입고 처리** — 오늘 입고예정인 생산발주가 있으면 재고에 더함
2. **밀린 주문 처리** — 대기 중인 주문을 순서대로(FIFO) 재고가 되는 만큼 출고
3. **새 주문 처리** — 오늘 들어온 주문을 재고 있으면 즉시 출고, 없으면 대기열에 추가
4. **출고일/도착일 계산** — 오늘 출고된 주문의 도착예정일을 결정 (정상/지연/기한 내/기한 초과)
5. **하루 마감 계산** — 기말재고, 가용재고, 대기 물량 계산
6. **재주문 확인** — 가용재고가 재주문점 이하면 생산발주 (필요하면 같은 날 여러 번)

**가용재고 공식**

```
가용재고 = 기말재고 + 아직 입고 안 된 생산발주 합 − 대기 중인 주문 합
```

**제품 상태 분류**

| 조건 | 상태 |
|---|---|
| 가용재고 < 0 | 부족 |
| 현재고 ≤ 재주문점 | 발주중 |
| 현재고 ≤ 재주문점 × 1.2 | 주의 |
| 그 외 | 정상 |

**배송 상태 분류**

| 조건 | 상태 |
|---|---|
| 희망배송일 없음, 주문 당일 출고 | 정상 배송 |
| 희망배송일 없음, 당일 이후 출고 | 배송 지연 |
| 희망배송일 있음, 목표 출발일 안에 출고 | 기한 내 배송 |
| 희망배송일 있음, 목표 출발일을 넘겨 출고 | 기한 초과 |

(목표 출발일 = 희망배송일 − 배송 리드타임)

화면은 이 계산 결과를 그대로 보여주되, 기준일 시점에서 다시 해석합니다. 예를 들어 기준일 이후에 입고될 생산발주는 "입고 예정", 기준일 이후에 출고될 주문은 "대기 중"으로 보여줍니다.

## 더 개선하고 싶었던 것

### 재주문점 · 안전재고 동적 재산정

가구는 시즌·트렌드 영향이 큰 소비재라 고정 재주문점은 한계가 있습니다. 최근 N일 재고 회전율과 수요 변동성을 기반으로 재주문점을 주기적으로 재산정하는 진단 레이어를 붙이려 했는데, 시뮬레이션 엔진 안정화를 우선했습니다. 지금 구조에서는 진단 결과를 화면에 "참고용 권장 재주문점"으로 추가하는 정도로 확장할 수 있습니다.

### 희망배송일 있는 주문의 예약 재고 우선 배정

재고가 부족할 때, 입고 시점까지 여유가 있는 예약 주문(희망배송일이 입고예정일보다 늦은 주문)의 재고를 먼저 급한 주문에 배정하는 방법을 검토했습니다. 재고 효율은 높아지지만 결국 먼저 주문한 고객의 재고를 나중 주문에 쓰는 셈이라 FIFO 원칙 훼손과 특정 고객 역차별 문제가 있어 채택하지 않았습니다. 운영자가 수동으로 승인하는 흐름을 붙이면 활용 여지가 있어 보입니다.

### 배송 용량(캐파) 모델링

같은 날 출고가 몰리면 자체 배송 차량으로 소화하고, 초과분은 외부 용역으로 처리하는 방식을 고려했습니다. 자체/외주 비용 차이를 반영하면 "오늘 배송 용역 비용이 얼마나 나오는지"도 보여줄 수 있는데, 이번에는 재고 흐름 계산에 집중하고 캐파 제약은 무제한으로 단순화했습니다.

### 묶음 배송

같은 고객이 같은 날 매트리스와 프레임을 함께 주문했다면 한 트럭에 같이 보내는 것이 현실적입니다. 지금은 주문 건별로 독립 처리하고 있어 이런 묶음 배송 시나리오는 다루지 못합니다. 제공된 데이터에 동일 고객의 중복 주문이 없어서 이번 구현 범위에서는 제외했습니다.

### 반품 처리

출고 후 반품이 들어오면 재고가 복귀되고 원장도 수정돼야 합니다. 지금은 출고 전 취소만 지원합니다. 반품 검수·재고 등록 워크플로까지 포함하면 도메인이 꽤 복잡해져서 우선순위에서 뒤로 뺐습니다.

### 주말·공휴일 캘린더

지금은 365일 운영으로 가정해 생산·배송 리드타임을 순수 일수로 계산합니다. 영업일 캘린더를 붙이면 실제 입고일·도착일이 다음 영업일로 밀리는 케이스를 정확히 처리할 수 있습니다.

### 프론트엔드 테스트 · API 통합 테스트

시뮬레이션 엔진 단위 테스트는 있지만, API 응답과 화면 동작을 검증하는 테스트가 없습니다.

### 페이지네이션

주문·원장 전체를 한 번에 불러오는 구조라 데이터가 많아지면 느려질 수 있습니다.
